# 부트캠프 공유 맥락 (Single Source of Truth)

> 모든 FT가 주제카드·데이터·MVP를 **같은 기준**으로 만들기 위한 단일 기준 문서다.
> 스킬(`topic-card-builder`, `tourism-data-validator`, `mvp-feasibility-tester`)은 모두 이 문서를 근거로 동작한다.
> 사실관계가 바뀌면 이 파일만 고치면 된다.

> ## ★ 기준(고정) vs 예시(시드) — 가장 먼저 읽을 것
> 이 문서의 내용은 두 종류다. 둘을 혼동하면 안 된다.
> - **기준(고정)**: 부트캠프 사실관계(§1), 학습 프레임(§2), 분석 규칙(§3), 카드 3층 스키마 *구조*(§4), 데이터 티어(§8), 도구 원칙(§9). → 방법·규칙. 잘 안 바뀐다.
> - **예시(시드)**: 처음 받은 자료에서 뽑은 **5개 지역, 6개 영역(A~F), 5개 시그니처(S1~S5), 12개 샘플 카드, 공급 수치**. → 이건 *정답표가 아니라 출발점*이다. **진짜 기준은 데이터에서 나온다.** 1·2번(수집·선정)을 돌리면 새 이슈·새 영역·새 시그니처·새 지역이 나올 수 있고, 그러면 예시는 **갱신·확장·교체**된다.
> 원칙: 스킬과 FT는 예시를 *문체·형식의 참고*로만 쓰고, **내용은 데이터로 다시 검증·발견**한다. "기존 5개/6개/12개 안에 맞춰야 한다"고 생각하지 않는다.

## 0. 두 국면 & Pre-부트캠프 파이프라인

작업은 **두 국면**으로 나뉜다. 도구와 협업 방식이 다르다.

- **Pre-부트캠프 (FT 준비)** — *우리(FT)가* 준비. **코워크/클로드**로 작업하고, 산출물은 **공유 작업 폴더**(이 `ft-skills/` 폴더)로 공유한다. (별도 GitHub/Slack은 쓰지 않음)
- **실제 부트캠프 (참가자 진행)** — 참가자가 2박 3일 바이브 코딩. **코덱스**에서 진행. 준비 단계에서 *확정한 산출물*(주제카드·정제 데이터/API·참가자 스킬)을 사용.
> 스킬을 `SKILL.md` 표준으로 만든 이유: 준비용은 코워크/클로드, 참가자용은 코덱스 — 같은 포맷이라 그대로 넘어간다.

### Pre-부트캠프 3단계 파이프라인
1. **데이터셋 수집** — 공공데이터포털·한국관광 데이터랩·통계청 SGIS·NIA AI허브(문화관광). 코워크로 실행하되 **수집·정제 표준(플레이북)** 을 따른다. 산출물은 원본 더미가 아니라 **카탈로그(인덱스) + 분석용 정제 테이블**.
   - **지역 범위 = 전국 최대 커버리지**(5개 검증 지역은 문체·시드 예시일 뿐, 스코프가 아님). 지역이 많아야 §3-A 정규화·갭 분석이 신뢰성 있게 작동한다.
   - TourAPI 공급(`areaBasedList2` totalCount)은 **전국 시군구 전수 자동 수집** 가능 → 정제 테이블의 백본. 방문자수 API는 전국 자동, 체류·소비는 CSV라 부분 보강.
2. **이슈×지역 선정(리스트업)** — 카탈로그를 입력으로, 데이터 기반 관광 이슈와 그 이슈가 나타나는 지역을 리스트업. **§3-A 정교한 기법**으로 추천도·운영 난이도·신뢰도 등 지표까지 산출(= 신뢰도 높은 카드 선정). → `topic-card-builder`의 입력.
3. **결과물 테스트** — 선정된 이슈로 "이슈 → 결과물"까지 가보며 **참가자가 쓸 스킬셋 자체를 FT가 먼저 돌려 검증·수정**한다. 즉 **3번의 대상은 산출물(프로토타입)이 아니라 *스킬셋*** 이다. 검증된 스킬셋을 다듬어 **그대로 실제 부트캠프(코덱스) 참가자 운영 스킬셋으로 승격**한다. (SKILL.md 동일 포맷이라 코워크/클로드 테스트 → 코덱스 운영으로 변환 없이 이동)
> ⏳ 단, **참가자 운영 스킬셋의 구체 설계는 1·2번(수집·선정)을 끝낸 뒤** 한다. 어떤 이슈·데이터가 확정되는지 보고 정한다.

> **인터페이스 = 카탈로그.** 1번이 카탈로그에 등록하고, 2번이 그걸 읽는다. 동료와는 원본 전체가 아니라 *카탈로그 한 파일*을 공유한다.
> **3번 = 테스트 스킬셋 → 참가자 운영 스킬셋.** 테스트는 "참가자가 쓸 스킬을 FT가 미리 깨보는" 과정이고, 산출물이 아니라 *스킬*을 승격한다(일부는 FT 전용으로 남김 → §4 층③).

## 2. 핵심 학습 프레임: Agentic Workflow

참가자는 "AI 딸깍"이 아니라 **사람이 판단하고 AI가 초안을 만드는 협업**을 훈련한다. 5단계 루프:

1. **문제 제시** — 맥락·목표·제약을 AI에게 제공
2. **AI 초안 생성** — 분석·아이디어·산출물 초안 생성
3. **결과 검토** — 체크리스트로 타당성·과장·누락 점검
4. **피드백** — 부족한 점을 구체적으로 지적하고 재요청
5. **개선** — 문제·데이터·솔루션·데모·피칭을 반복 개선

> **FT 함의**: 주제카드는 정답을 대신 만들어 주는 게 아니라 **첫 사고 흐름(scaffold)** 을 잡아주는 도구다. 카드가 솔루션까지 정해버리면 안 된다.

## 3. 분석 수준 규칙 (★중요★ — 이중 기준)

분석 깊이는 **누가 하느냐**에 따라 다르다. 둘을 섞지 않는다.

### 3-A. FT 선정용 (Pre-부트캠프 2번 단계) — 정교한 기법 허용
주제카드의 **신뢰도는 FT가 깊은 분석으로 보장**한다. 이슈×지역을 신뢰성 있게 선별하려면 단순 합계를 넘어선 기법을 쓴다:
- **지역 정규화**: 인구·면적·방문자 대비 z-score/지수화로 지역 간 공정 비교
- **추세·계절성**: 전년 동기 대비, 이동평균, 성수기/비수기 패턴
- **공급-수요 갭**: 수요 신호 대비 공급(장소 개수) 부족/과잉을 지수로
- **복합 지표·랭킹 합성**: 여러 지표를 가중합해 이슈 강도 산출
- **유사 지역 군집화**: 시그니처가 닮은 지역 그룹핑(→ §6, issue-signatures)
> 이 결과는 카드의 **FT 전용 메타**(신뢰도 점수·이슈 강도)로 들어가고, 참가자에게 그대로 노출하지 않는다.

### 3-B. 참가자용 (실제 부트캠프) — 단순 5가지만
참가자는 통계 전문가가 아니다. **다음 5가지 단순 분석만** 쓴다. t검정·회귀·군집은 쓰지 않는다.

| 분석 방법 | 사용 예시 | 참가자 표현 | 주의 |
|---|---|---|---|
| 평균 | 월평균 방문자 수, 월평균 관광소비 | "평균보다 높다/낮다" | 원인 단정 금지 |
| 합계 | 연간 방문자 수, 관광소비 총액 | "규모가 크다/작다" | 절대값보다 비교용 |
| 순위 | 인기 관광지 상위 10개, 맛집 상위 10개 | "어디에 몰려 있다" | 순위가 품질을 뜻하진 않음 |
| 비교 | 관광지 수 vs 숙박/음식점 수, A지역 vs B지역 | "공급이 불균형하다" | 단서 수준으로만 사용 |
| 증감 | 전년 동월 대비, 전월 대비 | "최근 늘었다/줄었다" | 일시적 이벤트 확인 필요 |

**표현 원칙**: 데이터는 *단서*다. "~때문이다"(원인 단정)가 아니라 "~해 보인다 / ~할 가능성이 있다 / ~가 필요해 보인다"로 쓴다.

> 핵심: **FT는 깊게 분석해 신뢰도 높은 카드를 만들고, 참가자는 그 위에서 단순 분석으로 검증을 경험**한다.

## 4. 주제카드 구조 (이슈 중심 · 3층 스키마)

> ★ 운영 방침(2026-06 회의):
> ① 카드는 **특정 지역에 고정하지 않는다.** 중심은 *데이터로 확인되는 관광 이슈*, 지역은 후보로만.
> ② 카드는 **3개 층**으로 구성한다. 누가·언제 보는지로 나눈다. 참가자가 보는 면은 가볍게, 풍부함은 FT 메타로 민다.
> ③ 요소를 추가할 땐 두 기준을 통과해야 한다: **(a) 정답을 주지 않고 사고를 비계(scaffold)하는가, (b) 피칭 평가지표(문제정의25·기획력25·실현30·기술20)에 연결되는가.**

각 요소에는 태그를 단다: `[필수]` 모든 카드·항상 노출 / `[선택]` 팀 상황 따라 차등 제공 / `[FT]` 참가자 비노출·선정 근거.

### 4.1 층 ① 핵심 (필수 — 모든 참가자, 항상, 이 순서)
1. `[필수]` **문제 중심 주제(이슈)** — 지역명 없이 이슈 자체를 질문으로.
2. `[필수]` **대상 사용자/페르소나** — 누가 겪는 문제인가(외국인·차 없는 여행자·야간 방문객 등) + 현장 시나리오 한 줄. (문제정의 25점 직결)
3. `[필수]` **검증 전 가설** — 검증 대상으로 둠. 단정 금지.
4. `[필수]` **활용 데이터 및 분석 계획** — 데이터랩(수요)+TourAPI(공급), 어느 지역에나 같은 지표로. 참가자 분석은 §3-B 단순 5가지.
5. `[필수]` **확인하고 싶은 단서** — 무엇이 보이면 가설이 힘을 얻는지(= 이슈 데이터 시그니처, §6).
6. `[필수]` **예상 아웃풋 이미지** — 발표 가능한 데모 화면의 최소 형태.
7. `[필수]` **추천 MVP** — 입력 → 처리 → 출력 → 발표화면 한 줄 (2박 3일 내).

### 4.2 층 ② 펼치는 층 (선택 — 팀 속도에 맞춰 차등 제공)
카드 본문엔 접어두고 FT가 상황에 맞게 꺼낸다.
- `[선택]` **막힐 때 힌트(Support)** — 어려운 팀: 지표 2개로 좁히기, 다운로드 클릭 경로, 빈칸 채우기 문장, 가장 쉬운 MVP-.
- `[선택]` **추가 과제(Stretch)** — 빠른 팀: 지역 2곳 비교, 업종별 소비 쪼개기, RAG 장소 설명, 외국어/접근성 옵션.
- `[선택]` **접근 방식 힌트** — 솔루션 유형 후보(지도·챗봇·추천·대시보드). **"데이터 본 뒤에만"** 조건 — 정답이 되지 않게.
- `[선택]` **흔한 함정/안티패턴** — "이렇게 하면 망한다" 1~2개.
- `[선택]` **유사 사례/레퍼런스** — 다른 지역의 비슷한 서비스(영감용).
- `[선택]` **단계별 체크포인트** — Day1/Day2 중간 산출물 기준(페이스 조절).
- `[필수★]` **후보 지역 선택 테이블** — ★참가자 자율성의 핵심. 카드는 **지역을 정해주지 않고**, 데이터로 검증된 **후보 지역군 + 각 후보의 특징(공급 수치·인구·인구감소·성격)** 을 표로 준다. 참가자가 근거를 갖고 직접 고른다. (region-issue-scout가 생성) 데모 예: `pipeline/cards/참가자카드_예시_인구감소_문화체류전환.md`
  - 표의 공급·인구 특징은 TourAPI/통계 실측, **체류시간·소비 등 수요 특징은 "검증 시 수집"**(고른 지역만 데이터랩 보강).
- `[선택]` **유사 이슈 지역** — 위 후보 테이블에 흡수. (별도로 두지 않음)

## 6. 이슈 데이터 시그니처 & 후보 지역 — 시드 예시

> ⚠ **시드 예시.** 아래 5개 시그니처(S1~S5)와 예시 지역은 처음 자료 기준의 *출발점*이다. 1·2번에서 전국 데이터를 보면 **새 시그니처가 발견되거나 기존 게 바뀔 수 있다.** 이 5개에 갇히지 말 것.
> 지역은 고정 배정이 아니다. 각 이슈는 데이터로 알아볼 수 있는 **시그니처(신호 패턴)** 가 있고, 그 시그니처를 보이는 지역이면 어디든 후보가 된다. 자세한 시그니처→유사 지역 찾는 법은 `issue-signatures.md` 참조.

### 이슈 유형별 데이터 시그니처 (요약)
| 이슈 유형 | 데이터 시그니처 | 예시 지역(검증됨) |
|---|---|---|
| 체류·소비 미연결 | 방문↑ + 평균 체류시간↓ + 방문 대비 관광소비↓ | 강릉 |
| 인구감소·문화 체류전환 | 인구감소지역 지정 + 문화시설 보유 + 당일 위주 | 안동 |
| 교통·접근성(분산형) | 자연·레포츠 공급↑ + 숙박·음식↓ + 관광지 분산 | 영월 |
| 외래객 언어·로컬경험 | 외래객 증감↑ + 음식·문화 콘텐츠 보유 | 전주 |
| 야간 체류·상권 연결약함 | 야간 방문 신호 + 야간 숙박·식사 연결↓ | 목포 |

> 위 5개는 **예시이자 문체 기준**일 뿐, 카드를 이 지역에 묶지 않는다. 같은 시그니처면 여수·통영·군산·남원·정선 등 다른 지역도 동일 카드의 후보다.

### 난이도 추정 기준 (지역 무관)
- **쉬움**: 데이터가 풍부하고 시그니처가 직관적, 단일 지역·단일 지표로 단서가 보임.
- **보통**: 수요+공급 2종 데이터를 비교해야 하거나 framing(인구감소·교통)이 필요.
- **어려움**: 다지역 비교·동선 분석·다중 데이터 결합이 필요.

### 운영 추천 매칭 (팀 속도 기준, 지역 아님)
- 초보팀 → **데이터 풍부+쉬움 시그니처**(체류·소비, 외래객 콘텐츠) 카드부터.
- 문제의식 강한 팀 → **인구감소·생활인구 framing** 카드.
- 발표 이미지 중요한 팀 → **야간·시각적 소재**가 있는 카드.

## 7. 가설→데이터 탐색 6단계 (참가자 진행 흐름)

FT가 참가자를 코칭할 때 따르는 흐름. (Day 1 핵심)

1. **현상을 한 문장으로 쓴다** — 단정 금지, "그럴 것 같다"로 둔다 → *문제 문장 초안*
2. **확인할 지표를 고른다** — 2~4개만. 많으면 Day 1에서 멈춘다 → *필요 데이터 목록*
3. **데이터랩에서 수요·소비 단서를 찾는다** — 월평균·합계·상위10·지역비교만 → *간단 표/스크린샷*
4. **TourAPI/공공데이터로 공급 단서를 붙인다** — "추천할 장소 후보가 있나" → *장소 후보 목록*
5. **단서를 문제 문장으로 바꾼다** — 원인 단정 금지, "어려워 보인다" 수준 → *검증된 문제 가설*
6. **솔루션은 마지막에 붙인다** — 데이터를 본 뒤에야 솔루션 제목 → *예상 아웃풋 이미지*

> 데이터를 보기 *전에* 솔루션부터 정하는 것이 가장 흔한 실패다. FT는 이 순서를 지키게 한다.

## 8. 데이터 소스 티어 (★표준★)

새 카드·검증의 데이터 출처는 아래 티어를 따른다. 자세한 링크는 `data-catalog.md`.

- **기본 (필수 활용)**: **한국관광 데이터랩** — 모든 카드는 데이터랩 신호를 최소 1개 이상 근거로 쓴다.
- **심화 (선택)**: **공공데이터포털**, **통계청 SGIS**(생활인구·인구·사업체 등 공간통계), **NIA AI 허브 — 문화관광 탭**(AI 학습용 데이터셋). 빠른 팀의 추가 과제나 공급·맥락 보강에 쓴다.
- **통합 접근**: **국내 공공데이터포털 API 키 1개**로 대부분의 공공 API(TourAPI·문화공연 등)에 통합 접근 가능 → 참가자에게 키 1개 발급을 안내한다.

> 원칙: 기본(데이터랩)으로 *수요·소비 신호*를 먼저 잡고, 심화 소스는 *공급·맥락 보강* 또는 *추가 과제*에만 쓴다. 처음부터 심화 소스로 시작해 Day 1에 멈추지 않게 한다.
